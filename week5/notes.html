JavaScript: Novice to Ninja Chapter 10: Testing and Debugging

Errors, Exceptions, and Warnings

  • Errors are cause when something goes wrong in a program, whether a system error, programmer error, or user error.
  • System errors mean there is a problem with the system or external devices with which the program is interacting. This can be difficult to fix but we should be aware of them and attempt to reduce their impact by working around any problems they cause.
  • Programmer errors are our responsibility and we must minimize and fix them - they could be incorrect syntax, faulty logic, or typos.
  • User errors could be considered programmer errors so we should try to predict any possible interactions that might throw an error and provide a solution that doens't negatively affect the user experience.
  • Exceptions are errors that produce a return value that can then be used by the program to deal with the error.
  • An exception will also produce a stack trace, which is a sequence of functions or method calls that lead to the point where the error occured. It will work backwards from point where the error occurred to find the original function or method that started the sequence.
  • Warnings can occur if there's an error in the code that isn't enough to cause the program to crash. The program will continue to run after the warning but the issue that produced the warning may cause the program to continue running incorrectly.

The Importance of Testing and Debugging

  • Errors that fail silently in the background make them difficult to spot and longer to track down.
  • Ensure that code in development fails loudly so that errors can be identified and fixed quickly.
  • Any errors in code that is in productions should fail gracefully so that the user experience is not affected, if possible.
  • Make sure exceptions a caught and dealt with and code is tested rigorously.

Strict Mode

  • Strict mode produces more exceptions and warnings and prohibits the use of some deprecated features.
  • Helps spot more errors early on.
  • Writing in strict mode can help improve code's clarity and speed.
  • Encourages a better quality of JavaScript to be written, so its use is recommended
  • To use strict mode, add 'use strict'; to the first line of a JavaScript file.
  • The recommended way to invode strict mode is to place all your code into a self-invoking function. This is especially helpful when using someone else's code so that only your code is forced to use strict mode.
  • Modules are in strict mode by default.
  • Linting tools can be used to test the quality of JavaScript code beyond strict mode. They are useful for enforcing a programming style guide.

Feature Detection

  • The recommended way to determine browser support for a feature is to use feature detection. This is done by using an if statement to check whether an object or method exists before trying to actually call the method.
  • Feature detection guarantees that the method is only called if it actually exists and fails gracefully, without any exceptions being thrown, if the method doesn't exist.

Debugging in the Browser

  • Debugging is the process of finding out where bugs occur in the code and then dealing with them. Errors often originate in a different place than when they occur.
  • Breakpoints can be created to halt the progress of the code so we can view the value of different variables at that point in the program.
  • You can use the alert() method to show a dialog at certain points in the code to check the value of variables at those points. However, this is now discouraged since new debugging tools have been developed.
  • console.log() can also be used. It won't stop the execution of the program like alert() but can be used to log the value of variables at different stages of the porgram.
  • console.trace() will log an interactive stack trace in the console.
  • The debugger keyword can be used inside your code to create breakpoints to pause the execution of the code. Just make sure to remove any references to the debugger command before deployment of code!

Error Objects

  • An error object can be created by the host environment when an exception occurs, or it can be created in the code using a constructor function. The parameter is used as the error message.
  • There are several error objects that can be used, including: Error, RangeError, ReferenceError, SyntaxError, TypeError, URIError, and InternalError.
  • These error objects can also be used as constructors to create custome error objects.
  • It is also possible to throw your own exceptions using the throw statement. This will allow for any problems in your code to be highlighted and dealt with.

Exception Handling

  • Exception handling allows you to handle exceptions gracefully by catching the error. Then errors can be hidden from user, but still identified. The error can then be dealt with while keeping the program running.
  • For pieces of code that may result in an exception, we can wrap them in a try block.
  • If an exception occurs, it will pass the error object that is thrown into a catch block.
  • The code inside the catch block will only run if an exception is thrown inside the try block.
  • A finally block can be added after the catch block that will always be executed after the try or catch block whether an exception occurred or not.

Tests

  • Testing is an important part of programming that can often be overlooked.
  • Test-driven development (TDD) is the process of writing tests before any actual code. Then writing some code to make the tests pass.
  • Here is a workflow for TDD:
    1. Write tests (that initially fail)
    2. Write code to pass the tests
    3. Refactor the code
    4. Test refactored code
    5. Write more tests for new features
  • Test-driven development is considered to be best practice but isn't always used. Any tests are better than no tests.
  • Testing frameworks like Jest provide a structure to write meaningful tests and then run them.
