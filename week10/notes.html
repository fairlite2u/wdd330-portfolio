MDN Web Docs: Client-Side Form Validation

What is form validation?

  • The browser and/or the web server checking to see that the data entered in the form is in the correct format and within the constraints set by the application.
  • Validation done in the browser is called client-side validation, while validation done on the server is called server-side validation.
  • Why should forms be validated?
    • To get the right data, in the right format.
    • To protect user's data.
    • To protect ourselves.

Using built-in HTML5 form validation

  • Most user data can be validated without relying on JavaScript using HTML5 form controls like required, minlength and maxlength, min and max, type, and pattern.
  • If the data follows all the rules specified and validates, the browser will submit the form. If anything is still invalid, the browser will block the form and display an error message.
  • CSS can be used to change styling on form fields that aren't valid with the :invalid pseudo-class.

Validating forms using JavaScript

  • Using JavaScript for validation allows you to take control over the look and feel of native error messages and deal with legacy browsers.
  • The Constraint Validation API consists of a set of methods and properties available on some form element DOM interfaces and make it easier to customize error messages.
  • You can also validate form without a built-in API by writing your own JavaScript. In these situations, you should ask yourself a few questions:
    • What kind of validation should I perform?
    • What should I do if the form doesn't validate?
    • How can I help the user to correct invalid data?
  • When it comes to client-side form validation, always remember to think carefully about the user
    • Display explicit error messages.
    • Be permissive about the input format.
    • Point out exactly where the error occurs, expecially on large forms.

MDN Web Docs: Using Fetch

  • The Fetch API provides a JavaScript interface for accession and manipulation parts of the HTTP pipeline, such as request and responses.
  • The simplest use of fetch() takes one argument — the path to the resource you want to fetch — and returns a promise containing the response.
  • Fetch requests are controlled by the connect-src directive of Content Security Policy rather than the directive of the resources it's retrieving.
  • The fetch() method can optionally accept a second parameter, an init object that allows you to control a number of different settings, including: method, headers, body, mode, credentials and redirect.
  • To cause browsers to send a request with credentials included on both same-origin and cross-origin calls, add credentials: 'include' to the init object.
  • To only send credentials on the same origin, add credentials: 'same-origin'.
  • To not include credentials, use credentials: 'omit'.
  • The Headers interface allows you to create your own headers object via the Headers() constructor.
  • Response instances are returned when fetch() promises are resolved.
  • The most common response properties you'll use are: Response.status, Response.statusText, and Response.ok.
  • Bothe request and responses may contain body data. A body is a instance of any of the following types:
    • ArrayBuffer
    • ArrayBufferView
    • Blob/File
    • string
    • URLSearchParams
    • FormData
  • Both request and response will try to intelligently determine the content type.
  • Fetch API support can be detected by checking for the existence of Headers, Request, Response, or fetch() on the Window or Worker scope.
